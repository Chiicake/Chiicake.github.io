# zh-reduce-ai Skill 设计文档

## 目标

设计并安装一个面向 OpenCode 的全局 skill，用于中文技术文本改写，核心目标是在不破坏技术信息的前提下降低 AI 味、削弱模板腔，并把改写风格控制在“轻度解释性、轻度口语化、但不失专业”的范围内。

该 skill 不是通用中文写作器，而是一个受约束的 rewrite skill。它应当优先服务于以下场景：

- 已经存在中文原文；
- 原文是论文、技术文档、设计说明、技术博客、实现说明等偏技术文本；
- 用户希望在保留原始技术事实的前提下，调整文风，降低机械感、模板感和明显的 AI 腔。

## 使用边界

### 适用场景

这个 skill 只在“中文技术改写”场景下触发，尤其适用于：

- 论文段落改写
- 技术设计文档改写
- 技术博客段落润色
- 实现说明、接口说明、部署说明的文风调整
- 需要根据给定风格样本，对中文技术文本做保守改写

### 不适用场景

以下场景不应触发该 skill：

- 从零生成长文
- 纯营销文案、品牌文案、广告文案
- 文学性润色或情绪化表达
- 非技术语境下的自由创作
- 不存在原文、只有主题或提纲的写作请求

这一步很关键。因为用户提供的原始 prompt 本质上是“给定原文 -> 产出改写后文本”的约束性说明，而不是开放式写作要求。

## 核心设计原则

### 一、信息守恒优先于风格模仿

skill 的最高优先级应当是信息守恒。

任何改写都不能破坏以下内容：

- 技术术语
- 库名、框架名、协议名
- 代码片段
- 文件名、类名、函数名、模块名
- 配置项、路径、API 路径
- 逻辑关系、因果关系、条件关系
- 事实判断和功能描述

换句话说，这个 skill 的任务不是“写得更像人”，而是“在信息不变的前提下，去掉明显的 AI 写作痕迹”。

### 二、字数控制是硬约束，不是软建议

用户原始 prompt 多次强调：输出不应多于原文，长度应与原文相符。

因此，skill 应把长度控制写成明确规则：

- 默认目标是“等长或略短”
- 禁止为了去 AI 味而大幅扩句
- 如需加入轻微解释，必须通过压缩其他位置来抵消字数增长
- 优先在局部词组层面替换，不优先对整句做扩写式改造
- 若因标点、括号整合、代码标识嵌入导致完全等长不可行，只允许出现可忽略的微小波动；总体仍然必须控制在“视觉上等长或更短”的范围内

这一点是 skill 成败的关键。很多“去 AI 味”改写失败，不是因为风格没变，而是因为字数无控制地膨胀。

### 三、风格要降低 AI 味，但不能滑向低质量口语

用户给定的 prompt 要求“更通俗、稍微啰嗦一点、略具解释性”，但同时明确禁止：

- 过于口语化
- `xxx呢` 这种句式
- 第一人称

因此，该 skill 的目标风格应定义为：

- 句子更舒展，但不过分拉长
- 允许少量解释性连接词
- 允许轻度动作展开，例如“管理”改成“进行管理”
- 不使用聊天口气、感叹句、情绪化词汇
- 不使用“至于 X 呢”“我们可以看到”“我认为”这类高风险句式

## 推荐 skill 名与安装位置

### 名称

推荐 skill 名：`zh-reduce-ai`

原因：

- 满足 OpenCode 对 skill name 的格式要求：小写、字母数字、连字符
- 语义清楚：中文 + 技术 + 改写
- 适合在全局 skills 列表中被准确检索

### 安装位置

推荐安装到一个规范主路径：

1. `~/.config/opencode/skills/zh-reduce-ai/SKILL.md`

这是 OpenCode 官方全局路径，也是本次实现的唯一 canonical deliverable path。

如需兼容当前环境中的 `.agents/skills` 发现链路，兼容策略定义为：

- 可选创建软链接：`~/.agents/skills/zh-reduce-ai -> ~/.config/opencode/skills/zh-reduce-ai`
- 不做两份独立内容副本
- 如果软链接不方便创建，则本次实现以 OpenCode 官方路径为准，不再追加第二份物理安装

这样处理的目的有两点：

- 避免重复发现与内容漂移
- 保持全局 skill 的单一事实来源

## frontmatter 设计

### name

`zh-reduce-ai`

### description

description 需要写得足够窄，避免误触发。推荐方向：

`Use when revising existing Chinese technical prose that must keep original terminology, logic, and near-original length.`

这个 description 的重点不是解释 skill 做什么，而是明确触发条件：

- 现成中文原文
- 技术文本
- 改写任务
- 必须保留术语和逻辑
- 需要接近原始长度

## skill 内容结构设计

建议 `SKILL.md` 采用以下结构：

1. `Overview`
2. `When to Use`
3. `Hard Constraints`
4. `Rewrite Moves`
5. `Forbidden Patterns`
6. `Length Control`
7. `Bracket and Identifier Handling`
8. `Quality Check Before Delivering`
9. `Mini Examples`

### 1. Overview

说明这不是自由写作 skill，而是中文技术改写 skill。强调核心原则：

- 先保真，再修风格
- 先控长度，再做润色
- 去 AI 味，不去技术性

### 2. When to Use

明确触发条件：

- 已有中文原文
- 文本包含技术术语、路径、代码标识、配置项等
- 用户要求改写、润色、降低 AI 味、改得更自然
- 用户不希望事实变化或字数膨胀

并明确写出 `When NOT to use`，防止模型把它用在自由创作上。

### 3. Hard Constraints

这一节应是整个 skill 的核心，建议直接写成硬规则：

- 不改变技术术语
- 不改变代码和标识符
- 不改变逻辑和因果
- 不用第一人称
- 不出现 `xxx呢`
- 输出不得长于原文
- 禁止用空泛套话填充篇幅

### 4. Rewrite Moves

把用户给的 rewrite 手法整理成一组“可用但受限”的动作，而不是机械替换器。

应包含：

- 动词短语轻度展开
- 局部加辅助词让句子更顺
- 常见词替换表
- 书面表达向轻解释性表达转换
- 条件句式适度放松，但不转成过度口语

同时需要明确：这些动作是候选动作，不是对每个句子都强制执行。

### 5. Forbidden Patterns

需要把高 AI 味和低质量口语两个方向一起拦住。

建议列出两类禁用项：

第一类：AI 味高风险模板

- 空泛开头
- 机械的“首先/其次/最后”堆叠
- 抽象口号式总结
- 无信息增量的过渡句
- 大量“能够有效地”“具有重要意义”之类空泛表达

第二类：低质量口语化

- `xxx呢`
- 第一人称
- “其实”“就是说”“大家都知道”这类聊天腔
- 过度解释导致句子失去技术密度

### 6. Length Control

这一节必须单列，不应埋在其他章节里。

需要给模型一个明确的操作顺序：

1. 先找可删的套话
2. 再做局部词替换
3. 只有在字数仍受控时才允许少量展开
4. 一旦句子变长，就在其他位置压缩回去

这能显著降低“越改越长”的概率。

### 7. Bracket and Identifier Handling

这一节吸收用户 prompt 里关于括号处理的规则：

- 解释性括号优先自然整合
- 如果整合后太拖沓，可谨慎省略非核心解释
- 代码、路径、类名附近的括号优先转成自然并列而不是保留原状

同时要强调：任何涉及技术标识的整合都不能改动原字面值。

### 8. Quality Check Before Delivering

建议做成简短 checklist：

- 技术术语是否原样保留
- 逻辑是否未变
- 是否出现第一人称
- 是否出现 `xxx呢`
- 是否比原文更长
- 是否引入了空泛套话
- 是否比原文更像机械模板

### 9. Mini Examples

这一节只放少量、短小的前后对照，不放大段示例，以免模型过度模仿单个句子壳子。

示例应覆盖：

- 动词短语展开
- 括号整合
- 术语保留
- 长度受控的改写

## 对原始 prompt 的优化策略

用户给的 prompt 已经很完整，但如果直接照搬进 skill，会有三个问题：

### 问题一：机械替换感太强

如果直接把“采用 -> 运用”“通过 -> 借助”写成硬替换，模型可能会在不合适的语境里滥用这些词，反而制造新的模板腔。

优化方式：

- 把这些规则写成“优先替换候选”而不是无脑替换表
- 强调“以不变扭为前提”
- 强调“优先保持句意自然”

### 问题二：原 prompt 容易让模型误以为“越啰嗦越像人”

事实上，很多低 AI 味文本并不啰嗦，而是表达具体、局部解释自然。

优化方式：

- 把“稍微啰嗦”重新定义成“局部更舒展、但总字数不增长”
- 引导模型做微调而不是大扩写

### 问题三：原 prompt 对“AI 味”的负面模式定义还不够系统

优化方式：

- 在 skill 中新增“高风险 AI 模板”黑名单
- 让模型不仅知道怎么改，还知道哪些东西必须删

## 为什么应当做成全局 skill

这不是仓库私有规则，而是一类跨项目重复出现的任务模式：

- 技术博客成稿前的段落改写
- 技术设计文档的中文润色
- 论文、课程报告、实现说明中的去模板化改写
- 保留术语与路径的技术文本重写

这些任务在不同仓库、不同项目里都可能出现，而且共同特点稳定：有原文、有技术约束、有长度限制、有去 AI 味诉求。因此，把它做成全局 skill 是合理的；它不依赖某个单一项目的术语表、文件结构或品牌口吻。

## 全局安装后的预期行为

安装后，这个 skill 应当在以下请求中更容易被发现：

- “把这段中文技术文档改得自然一点”
- “帮我重写这段，别那么 AI”
- “保持技术准确，改写成更像人写的中文”
- “不要改术语，帮我润色这段技术说明”

它不应在以下请求中轻易触发：

- “帮我写一篇技术博客”
- “帮我做中文营销文案”
- “帮我起标题”
- “写一个产品介绍”

为避免实现阶段再度解释，这里给出应直接转写到 `SKILL.md` 的最小骨架。

### frontmatter

name: zh-reduce-ai

### When to Use

应直接包含这些意思：

- 输入已经是中文原文
- 文本属于技术说明、论文、技术博客、设计文档、实现说明等技术语境

应直接包含这些意思：

- 不用于从零生成文章
- 不用于营销文案或品牌文案

- 不得用空泛套话填充篇幅

## 风险与检查项

### 风险

- 规则写得过满，导致 skill 像替换字典而不是写作指导
- 触发范围过宽，误伤通用写作场景
- 对“去 AI 味”的追求压过了技术保真
- 字数控制没有被提升为真正硬约束

### 检查项

- skill 名是否符合 OpenCode 命名规则
- description 是否准确收窄触发条件
- 是否明确“不用于从零生成内容”
- 是否把长度控制写成硬规则
- 是否明确禁止第一人称与 `xxx呢`
- 是否对术语、路径、标识符保持零改动要求
