## 概述

使用 LLM 辅助生成前端页面时，一个常见问题并不是页面无法运行，而是结果往往带有稳定的模板化倾向：字体选择重复，色彩策略趋同，区块层级不清，卡片式布局泛滥，文案也容易停留在通用占位表达。

Impeccable 是一个面向 AI harness 的前端设计技能包。它建立在 Anthropic 原版 `frontend-design` skill 的基础上，增加了更细的设计参考、一组可直接调用的设计命令，以及用于限制低质量输出的反模式约束。它不替代前端框架，也不替代设计系统，而是试图让 AI 在处理页面设计任务时具备更明确的设计语言和修正路径。

下面从它解决的问题、命令组织方式和在当前项目中的使用方式几个角度来看 Impeccable。

## 它要解决的问题

AI 生成前端页面时，常见结果通常“可以工作”，但不一定有稳定的视觉质量。比较典型的问题包括：

- 字体和排版策略缺少区分度，不同页面类型经常共用同一套表达。
- 色彩选择偏向高频模板，视觉层级不稳定。
- 页面布局依赖卡片堆叠，区块关系不够清晰。
- 留白、动效与交互反馈存在，但缺少明确节奏。
- 标题、按钮和提示文本容易停留在泛化表达。

这些问题本身不一定造成功能错误，但会使页面呈现出明显的“AI 生成痕迹”。Impeccable 的思路不是要求模型抽象地“做得更好看”，而是把一部分前端设计知识拆解为更明确的设计维度、设计命令和反模式约束，使模型更容易判断应该修正什么，以及应当如何修正。

## 基本组成

从实际结构上看，Impeccable 主要由三部分组成：

1. 增强版 `frontend-design`
2. 可显式调用的设计命令
3. 一套反模式约束

### 增强版 `frontend-design`

Impeccable 的核心仍然是 `frontend-design`，但它进一步补充了多个设计参考方向。它把前端设计问题拆成了几个比较明确的方向：

- typography
- color and contrast
- spatial design
- motion design
- interaction design
- responsive design
- ux writing

这些维度覆盖的是前端设计中最常见、但又最容易被模糊处理的问题。

例如，排版不只是把标题放大，而是涉及字体搭配、字号层级、比例系统和阅读节奏；颜色也不只是选择主色，还涉及中性色组织、对比度控制和整体一致性；响应式并不只是让页面缩放，而是重新处理布局密度、区块顺序和内容优先级。

这种拆分的意义在于，AI 在处理设计任务时，不再只接收一个抽象目标，而可以从更具体的维度理解页面问题。

### 命令体系

Impeccable 提供了一组可以直接调用的命令。主要命令包括：

- `/teach-impeccable`
- `/audit`
- `/critique`
- `/normalize`
- `/polish`
- `/distill`
- `/clarify`
- `/optimize`
- `/harden`
- `/animate`
- `/colorize`
- `/bolder`
- `/quieter`
- `/delight`
- `/extract`
- `/adapt`
- `/onboard`
- `/typeset`
- `/arrange`
- `/overdrive`

如果按实际使用习惯划分，这些命令大致可以分成四类。

#### 1. 诊断类

- `/audit`
- `/critique`

`/audit` 更偏技术质量检查，例如可访问性、响应式、性能和布局问题；`/critique` 更偏设计与体验层面的评审，例如层级是否清晰、表达是否准确、页面是否具有合适的视觉重心。

如果不知道应该先改哪里，这两类命令通常是比较合理的起点。

#### 2. 结构整理类

- `/normalize`
- `/arrange`
- `/extract`
- `/adapt`

这组命令主要用于处理一致性、布局关系和结构组织问题。

`/normalize` 更适合把页面往统一标准上收拢；`/arrange` 更关注区块关系、间距与节奏；`/extract` 偏向把重复设计抽出为可复用结构；`/adapt` 则更强调设备与场景适配。

#### 3. 视觉与排版类

- `/typeset`
- `/colorize`
- `/animate`
- `/bolder`
- `/quieter`
- `/delight`
- `/overdrive`

这组命令更直接面向视觉表达。

其中，`/typeset` 实用性较强，主要处理字体、层级和阅读节奏；`/colorize` 处理颜色组织；`/animate` 处理有目的的动效；`/bolder` 与 `/quieter` 分别对应增强和收敛视觉表达；`/delight` 偏向补充局部体验细节；`/overdrive` 则更偏展示性和实验性。

对于博客、文档页和专题页，`/typeset` 往往比高强度视觉特效更有实际价值。

#### 4. 完成度与可用性类

- `/polish`
- `/clarify`
- `/optimize`
- `/harden`
- `/distill`
- `/onboard`

这类命令更接近“把页面从可用推进到完整”。

`/polish` 适合做上线前最后一轮整理；`/clarify` 更适合处理文案与表达不清的问题；`/optimize` 用于性能方向的改进；`/harden` 处理错误态、边界情况和 i18n 等问题；`/distill` 用于压缩不必要的复杂度；`/onboard` 则聚焦引导与上手流程。

### 反模式约束

Impeccable 的一个特点是，它不仅提供正向指导，也明确给出负向约束。比较典型的反模式包括：

- 不要过度依赖高频默认字体；
- 不要在彩色背景上使用低对比度灰字；
- 不要大面积直接使用纯黑或纯灰；
- 不要把所有内容都装进卡片，尤其不要出现卡片套卡片；
- 不要使用已经显得陈旧的弹跳式 easing。

这些限制本身不能替代设计系统，但它们能够减少 AI 输出中最常见的一批低质量模式。对于依赖 AI 生成页面初稿的工作流来说，这类反模式约束和正向建议一样重要。

## `teach-impeccable` 与项目级设计上下文

在所有命令中，`/teach-impeccable` 值得单独讨论，因为它不是直接修改页面，而是建立项目级设计上下文。

这个命令会先扫描代码库，识别已有的技术栈、组件模式、样式变量、品牌资产和设计 token；然后再针对无法直接从仓库中推断出的部分向用户提问，例如目标用户、品牌气质、视觉偏好和可访问性要求。

这些信息最终会写入项目根目录下的 `.impeccable.md`。这个文件的意义在于，它把一次性 prompt 中的设计偏好沉淀为可复用的项目上下文。

对于长期依赖 AI 辅助前端迭代的项目来说，这一点较为重要。很多页面问题并不是单次生成失败，而是多轮修改时缺少统一的设计基线。`teach-impeccable` 实际上是在补这一层基础信息。

```bash

```

目前它是一个包含 21 个 skills 的技能包，其中既包括增强版 `frontend-design`，也包括 `audit`、`polish`、`typeset`、`arrange`、`teach-impeccable` 这类命令型 skills。

## 它与原版 `frontend-design` 的关系

Impeccable 并不是完全独立于 `frontend-design` 的另一套体系，而更像是对原版 skill 的扩展。

它可以看作是对 Anthropic 原版 `frontend-design` 的补强：保留原有方向，同时增加更细的设计参考、更明确的命令控制，以及更具体的反模式限制。

因此，一个比较合适的理解方式是：

- 原版 `frontend-design` 负责提供总体设计方向；
- Impeccable 负责把这些方向拆成更具体的设计维度、显式命令和反模式约束。

如果项目中已经存在 `frontend-design`，那么 Impeccable 的价值通常不在于“从无到有”，而在于进一步提升模型处理前端视觉质量问题的能力，并把设计修改组织成更清晰的工作流。

## 适用场景

从命令组织方式和实际用途来看，Impeccable 更适合以下几类场景。

### 营销页与展示页

这类页面通常更依赖视觉层级、排版节奏和首屏表达，因此 `/typeset`、`/arrange`、`/polish` 这类命令会比较直接。

### 内容页与博客页

内容型页面未必需要复杂视觉效果，但通常非常依赖稳定的排版系统、清晰的标题层级和合理的阅读节奏。相较于“做得更炫”，Impeccable 在这类页面上的价值更偏向“做得更清楚”。

### 已有页面的审计与整理

如果项目里已经有一批页面，但一致性不足、留白关系混乱、局部风格漂移，那么先用 `/audit` 定位问题，再用 `/normalize`、`/arrange` 和 `/polish` 做逐步修正，通常比整体重写更稳妥。

### 长期使用 AI 辅助前端迭代的项目

如果一个项目会频繁借助 AI 做页面微调，那么 `.impeccable.md` 这种项目级上下文会比一次性 prompt 更有持续价值。

## 不适合把它当成什么

Impeccable 的边界也比较明确。

首先，它不是通用工程技能包。它对 API 设计、测试策略、后端架构、构建流程和数据建模没有直接帮助。

其次，它也不是设计系统的替代品。如果项目已经有成熟的品牌规范、组件库和视觉标准，那么 Impeccable 更适合作为执行层的辅助工具，而不是规则来源本身。

最后，它并不保证“装上以后页面自然变好”。它提供的是更明确的设计工具和约束条件，最终结果仍然取决于项目上下文、现有代码结构，以及这些命令是否被用在合适的问题上。

## 一个较为稳妥的使用顺序

如果准备在项目里正式使用 Impeccable，一种比较稳妥的顺序是：

1. 先运行 `/teach-impeccable`，建立项目级设计上下文；
2. 对现有页面执行 `/audit`，确认问题分布；

![在当前博客项目中执行 audit](./assets/1_audit.png)

上图展示的是在当前博客项目中对目标页面执行 `audit` 的过程。对于这类内容型站点，`audit` 的作用通常不是立刻重写页面，而是先确认问题集中在哪些位置，例如标题层级、段落节奏、留白关系和局部组件的一致性。

3. 使用 `/normalize` 或 `/arrange` 处理一致性与布局问题；
4. 对内容密集页补充使用 `/typeset`；

![在当前博客项目中运行优化后的页面结果](./assets/2_运行优化后.png)

在完成针对性的调整之后，更值得关注的通常不是视觉风格是否被彻底改写，而是页面是否在阅读路径、信息层级和整体节奏上更加稳定。对于博客和文档类页面，这类改动往往比强动效或大幅度视觉重构更有价值。

5. 最后用 `/polish` 做上线前整理。

这种顺序的好处在于，它把“诊断、修正、打磨”分开处理，避免一开始就进入大范围重写。

## 总结

Impeccable 的主要价值，不在于提供一套固定的页面模板，而在于把一部分前端设计知识、设计命令和设计约束组织成一套更明确的 AI 工作流。

把这些部分合在一起看，Impeccable 有几个比较明确的特点：

- 它建立在 `frontend-design` 的基础上，而不是完全另起体系；
- 它提供了较完整的命令体系，便于把设计任务拆成不同阶段；
- 它通过反模式约束减少常见的模板化输出；
- 它支持项目级设计上下文持久化，而不仅依赖单轮 prompt；
- 它更适合前端视觉与体验优化，而不是通用开发能力增强。

如果一个项目会持续使用 AI 辅助生成 landing page、内容页或做界面 polish，那么 Impeccable 是一个值得尝试的补充工具；如果工作重点主要在后端逻辑、脚本工具或非视觉型任务，它的收益则不会那么直接。
