# 分布式系统的幂等性清单

幂等性经常被简化成一句话：“同一个请求重试多次，结果应该一样。”这句话没错，但落地时远远不够。

在真实系统里，你不仅要考虑返回值是否一致，还要考虑数据库写入、消息投递、扣费、发券和调用第三方这些副作用是否被重复执行。

## 1. 先定义幂等边界

第一步不是写代码，而是明确“什么算同一个请求”。

常见做法是引入幂等键（Idempotency Key），但这个键必须和业务语义绑定，例如：

- 同一个订单创建请求；
- 同一笔支付确认；
- 同一个租户下的一次资源初始化。

如果幂等键只是一个随机 UUID，而没有和业务实体建立约束，那么你只是把问题推迟了。

## 2. 幂等记录不能只存结果

很多实现会把“请求成功后的响应体”直接缓存起来，下次命中同一个 Key 就原样返回。这种方式在简单场景里有效，但复杂场景需要更多状态。

更稳妥的做法是维护一套状态机，例如：

- `processing`
- `succeeded`
- `failed-retryable`
- `failed-terminal`

这样做的价值在于：你可以区分“第一次正在执行”与“已经完成”，也能避免并发重试时同时推进多个副作用。

## 3. 数据写入与副作用要拆开

如果一个请求既要写数据库，又要发消息、调第三方或者发通知，就不能只靠数据库唯一索引解决全部问题。

数据库唯一约束可以保证主记录不重复，但无法天然约束外部副作用。

### 更可靠的组合

1. 用幂等键保护主业务写入。
2. 用事务内 Outbox 记录待发送事件。
3. 由异步投递器负责把 Outbox 推给消息系统或第三方。
4. 在消费者侧继续做消费幂等。

这样可以把“请求已提交”与“副作用已送达”分成两个可恢复阶段。

## 4. 并发冲突是幂等设计的核心

最容易被低估的问题是：同一个幂等键在极短时间内同时打到两个实例。

如果你的实现流程是“先查有没有，再决定写不写”，那它几乎一定存在竞态条件。更好的方式包括：

- 利用唯一索引或条件写入原子创建幂等记录；
- 让后续请求看到 `processing` 状态时选择等待、轮询或快速失败；
- 为幂等记录设置超时恢复机制，防止实例崩溃后永久卡死。

## 5. 幂等记录也有生命周期

不是所有幂等键都要永久保存。不同业务可以有不同 TTL：

- 支付确认可能要保留几天；
- 接口防重可能只需要几分钟；
- 消息消费去重可能与消息保留期一致。

TTL 太短会导致重复副作用重新出现，TTL 太长会带来存储成本和错误命中。这个窗口需要根据业务补偿能力来定。

## 6. 一份实战清单

如果你正在给一个分布式写接口做幂等，我建议至少确认下面这些问题：

1. 幂等键是否和业务语义绑定。
2. 幂等记录是否有明确状态，而不只是缓存结果。
3. 业务写入和外部副作用是否被拆分。
4. 是否能处理同一幂等键的并发请求。
5. 是否定义了 TTL、异常恢复和人工排障方式。

幂等性不是“为了重试而重试”的附加功能，而是分布式系统里防止重复副作用扩散的基础设施。越早把它设计成状态机和流程问题，后面越不容易补锅。
