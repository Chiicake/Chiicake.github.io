原文链接：[What do people love about Rust?](https://blog.rust-lang.org/2025/12/19/what-do-people-love-about-rust/)

自从我们在 2015 年发布 1.0 以来，Rust 每年都被 Stack Overflow 评为“最受喜爱”（现在叫“最受推崇”）的语言。这意味着使用 Rust 的人还想继续使用 Rust[^gleam]--而且不只是拿它来做性能要求很高的工作或嵌入式开发，也会用它写 shell 脚本、Web 应用，以及各种你意想不到的东西。我们的一位参与者说得很到位：“到了这个阶段，除了 Rust，我已经不想用任何别的语言写代码了。”

[^gleam]: 2025 年，72% 的 Rust 用户表示他们想继续使用 Rust。过去，Rust 的分数曾经 *远远* 高于其他任何语言，但今年，[Gleam 差点就追上了](https://survey.stackoverflow.co/2025/technology/#admired-and-desired)，达到了 70%！真替他们高兴！Gleam 看起来很棒--而且，`fn` 这个关键字选得也很对。;)

当我们坐下来梳理愿景文档的数据时，我们特别想解释的一件事是：*究竟是什么激发了人们对 Rust 如此强烈的忠诚？*[^messitup] 根据访谈，答案既简单又复杂。简短的版本是，**Rust 让他们能够写出可靠且高效的软件**。如果这听起来很耳熟，那是应该的：[这正是我们官网上写着的口号](https://www.rust-lang.org)。更有意思的问题是，这种赋能究竟是**如何**产生的，以及这对我们如何演进 Rust 有什么启示。

[^messitup]: 还有，呃，我们怎么才能确保别把这件事搞砸？

## 人们欣赏 Rust 的哪些地方？

我们首先注意到的是，在每一场对话中，无论对方是在写第一个 Rust 程序，还是已经用了很多年；无论是在构建庞大的数据集群、嵌入式设备，还是只是随便折腾，他们提到喜欢 Rust 的点都高度一致。

第一点是 **可靠性**。人们喜欢那种“只要能编译，基本就能跑”的感觉：

> “我真正喜欢 Rust 的地方在于，只要它能编译，通常就能运行。这太棒了，而这是我在 Java 里不习惯的事情。” -- 汽车嵌入式系统高级软件工程师

> “Rust 就是那种会替你兜底的语言。你会睡得更好，而且你其实不需要那么‘聪明’。” -- Rust 顾问兼开源框架开发者

另一个当然是 **效率**。这一点在两个极端场景中特别常被提到：非常大规模（数据中心）和非常小规模（嵌入式）：

> “我想把机器资源留给[主要]计算，而不是拿去给看门狗偷走资源。” -- 从事数据科学平台工作的软件工程师

> “使用 Rust 还能带来速度收益。比如，[..] 仅仅是把这个 Python 组件换成 Rust 组件，就让我们的速度提升了 100 倍。” -- 某医疗设备初创公司的 Rust 开发者

当我们和运行 **“大规模”工作负载** 的用户交流时，“效率”尤其频繁地被提起，因为哪怕只是很小的性能提升，也可能转化为巨大的成本节省：

> “我们有一个库--本质上很像一个嵌入式数据库--部署在很多机器上。它原本是用 Java 写的，我们最近把它从 Java 重写成 Rust，性能大概提升了接近 9 到 10 倍。” -- 从事云基础设施服务的杰出工程师

> “在同一个模块里，加载 VM 的 Java 代码和 Rust 版本相比，我看到的是 4 倍效率提升。这能省下很多数据中心成本。” -- 专注于金融服务的后端工程公司创始人

在光谱的另一端，从事嵌入式开发或更底层抽象工作的受访者则强调 Rust 能提供 **底层控制力以及对系统细节的访问能力**：

> “Rust 就是我找了很多年、一直想用来替代 C 的那个语言。” -- 专注于金融服务的后端工程公司创始人

> “如果你要写新的东西，而且是在做这类偏底层、偏系统的工作，我觉得老实说 Rust 是唯一真正的选择。” -- 杰出工程师

很多人提到，Rust **支持性很强的工具链** 非常重要，它帮助他们快速上手，尤其是编译器的错误信息：

> “我认为我之所以能成功学会 Rust，很大一部分原因是工具链。对我来说，刚开始学 Rust 时，这门语言很有挑战，但工具链却异常容易上手。” -- 某开发者工具公司高管

> “这些工具对我个人有效，对我们团队也有效。我觉得自己与 Rust 打交道的首要方式，就是它的工具生态。我用 Cargo 构建代码，用 Cargo 测试代码。我们几乎所有事情都依赖 Clippy。” -- 从事安全关键型机器人工作的嵌入式系统工程师

> “我也觉得 Rust 编译器给出的报错和建议非常有帮助。” -- 专注于形式化验证的教授

最后，Rust 最重要的优点之一是它的 **可扩展性**。无论是语言本身，还是 crates.io 生态，Rust 都被设计成让终端用户能够创建满足自身需求的库和抽象：

> “crate 生态再加上稳定性保证和语义化版本控制，让它成了我见过最适合拿来就用的生态。” -- 计算机科学教授兼编程语言设计者

> “我觉得过程宏是 Rust 非常强大的一项超能力。” -- Rust 网络库的创建者与维护者

> “Rust 在让人非常非常容易上手、复用现有成果、快速试验新工具、新库以及其他各种东西方面，做得好得惊人……所以对我来说，作为一个实验平台，它非常棒。” -- 专注于嵌入式与实时系统的 Rust 专家兼顾问

# 但他们真正热爱的，是那种赋能感与通用性

可靠性、效率、工具链、生态系统--这些都是人们欣赏 Rust 的地方。但他们热爱的并不是其中任何单独一点，而是这些因素组合起来，使 Rust 成为一个**值得信赖、用途广泛的工具**，几乎可以被带去解决**任何问题**：

> “当我开始了解它时，我心里想的是：‘对，这就是我一直在找的语言。’这门语言会让我彻底不再去想用 C 和 Python。所以我就是得用 Rust，因为它能让我既能往下走到足够底层，也能往上走到足够高层。” -- 非洲的软件工程师兼社区组织者

> “我想要一门在整条技术栈里从上到下都很好用的语言，覆盖从嵌入式到非常复杂的应用。” -- 计算机科学教授兼编程语言设计者

> “如果 [Rust] 要以某种特定方式更多地推销自己，我大概会说：它是一门高性能、高表达力的通用语言，而且很棒的一点是，你可以用它把整条技术栈从上到下全都写掉。” -- 专注于嵌入式与实时系统的 Rust 专家兼顾问

## 每一部分都必不可少，整体才能成立

如果拿走可靠性，你就不会信任它：你会对每次部署都反复怀疑，害怕重构，也不敢让初级开发者碰关键路径。

> “Rust 就是把那道门槛降下来了。写出正确的 Rust 代码要容易得多。作为团队负责人，当经验没那么丰富的工程师参与这些关键应用时，我会安心很多。” -- 从事云基础设施服务的杰出工程师

> “我写 Rust 软件的体验往往是：**一旦你把它跑起来了，它就会一直保持可用**。这来自于语言在向后兼容性上投入的大量谨慎，以及整个生态在这方面投入的大量心力。” -- 专注于嵌入式与实时系统的 Rust 专家兼顾问

可靠性还提供了护栏，帮助人们进入新的领域--无论你是正在摸索门道的新手，还是涉足陌生领域的专家：

> “Rust 会把你带入所有这些东西，比如 `match`，还有这些非常棒的函数式编程方法。” -- 具有 Rust 生产环境经验的软件工程师

> “我认为 Rust 的所有权纪律对普通 Rust 程序员和做验证的人都很有用。我觉得它让你能够在函数作用域内非常清楚地知道：你修改了什么，什么没有被修改，什么发生了别名，什么没有发生别名。” -- 专注于形式化验证的教授

> “我发现了 Rust……基本上就是把它当成一种方式，让身为独立固件开发者的自己多一点信心。” -- 从事汽车数字驾驶舱系统的软件工程师

如果拿走效率和底层控制力，那就有些地方你去不了：嵌入式系统、实时应用，以及任何按周期计成本的场景。

> “Rust 的性能简直离谱。它好太多了，而且还安全。我们把 C++ 和 C 库，或者 C 应用重写成 Rust 后，结果往往会更快，因为 Rust 在内存布局上做得更好。” -- 主导消费者购物体验的高级首席工程师

> “十次里有九次，我写微控制器代码时只做单元测试。把它放到真实硬件上，第一次就能跑起来。” -- 从事安全关键型机器人工作的嵌入式系统工程师

> “我可以很有信心地构建可扩展的系统。” -- 拥有 20 年媒体与流媒体平台经验的工程经理

如果拿走工具链和生态，你就没法开始；或者就算能开始，也会走得异常艰难，而且你永远感觉不到高效。

> “对我来说，刚开始学 Rust 时，这门语言很有挑战，但工具链却异常容易上手……我可以直接开始写代码，它就能构建、能运行，这对我来说是巨大的区别。” -- 某开发者工具公司创始人兼 CEO

> “Cargo 是个惊人的包管理器。它大概是我用过最好的一个。我好像从来没在 Cargo 上碰到过问题。它就是能用。” -- 具有 Rust 生产环境经验的软件工程师

> “Rust 编译器给出的这类错误信息非常出色。它产出的错误类型对人帮助极大。而且不仅仅是报错本身，更重要的是，它还能抓住其他语言可能抓不到的错误。” -- 从事云基础设施服务的杰出工程师

## 结果：Rust 成为通往新领域的入口

当所有这些部分组合在一起时，会发生一件有意思的事：Rust 会变成一扇**入口**，通向那些原本难以触及的领域。我们听到了一个又一个故事：人们的职业道路之所以发生变化，是因为 Rust 给了他们信心，让他们敢于去做过去做不了的事：

> “我原来是土木工程专业，前端开发是自学的。我没有计算机背景。后来我对 Rust、分布式系统以及围绕它的设计和系统产生了兴趣。我改了专业，同时学习计算机科学和 Rust。” -- 正在转向密码学研究的软件工程师

> “过去 25 年里，我一直在 [某跨国工程技术公司] 的不同子公司工作，做的大多是 Java 领域的软件开发……两年前我开始接触汽车行业。在那个背景下，自然而然的结果要么是开始做 C++（而我并不想这样），要么就是抓住机会一头扎进刚刚建立起来的 Rust 生态。” -- 汽车嵌入式系统高级软件工程师

> “我最开始做区块链。现在白天的工作在做别的事情。Rust 实际上给了我进入那个领域的路径。” -- Rust 开发者兼航空航天社区领袖

> “在那之前，我用了 10 年一些动态编程语言做编程，尤其是 Ruby，用来开发 Web 应用。我想选择一门专注于系统编程的语言，所以我把 Rust 作为新的选择。这改变了我的职业生涯。” -- 从事汽车系统与区块链基础设施工作的 Rust 顾问兼作者

## 但这种平衡至关重要

Rust 的每一种特质，对它跨领域的通用性都不可或缺。但如果其中某一点走得太远，或者其他特质缺失了，它们也会变成障碍。

### 例子：复杂 API 与类型复杂度

Rust 最强大的方面之一，在于它的类型系统能够对应用领域的某些方面进行建模。这既能防止 bug，也能让新手更容易上手[^sleep]：

[^sleep]: ……对于那些睡眠不足、行为也确实很像新手的资深开发者来说，也是如此。

> “你不是只用原始位字段，而是有人把它编码进了类型系统。所以当你有一个像 'open door' 这样的函数时，如果门已经开着，你就不能再传一个 'open door' 进去。类型系统会直接把它踢掉并拒绝。” -- 从事汽车数字驾驶舱系统的软件工程师

> “你可以创建契约。比如，在哪些顺序下允许使用锁。” -- 从事汽车中间件开发的高级嵌入式系统工程师

但问题在于，有时为了把这些不变量编码进类型里所做的工作，会造出一种看起来比问题本身还复杂的东西：

> “当 Rust 同时涉及 async、泛型和生命周期时，那些类型会变得极其复杂，复杂到你基本得是某种 Rust 大神，才看得懂这段代码，或者有能力去写它。” -- 具有 Rust 生产环境经验的软件工程师

> “你不再是意大利面代码，而是意大利面类型。” -- 某汽车半导体公司的平台架构师

> “我觉得它更晦涩，也更难真正理解。类型描述的不只是这个东西的接口，还包括生命周期、你如何访问它、它是在栈上还是堆上，里面塞进了太多信息。” -- 从事数据科学平台工作的软件工程师

这让一些人主张，除非真的需要，否则不要使用 Rust 里某些更复杂的特性：

> “我的观点是，Rust 的难点--trait、lifetime 等--其实并不是提高生产力所必需的。完全可以通过安排学习曲线和库的方式，让人们快得多地完成入门。” -- Rust 网络库的创建者与维护者

### 例子：Async 生态性能很强，但还达不到“支持性”应有的标准

Async Rust 推动了 Rust 在构建网络系统上的巨大增长。但很多评论者都谈到一种感觉：`async Rust` 完全是比同步 Rust 更难的另一回事：

> “我觉得学习上先是一段爬坡，然后是一次跳跃，而 async 又在更远的那边。所以目标就是先让人对 Rust 有足够的兴奋感，好跨过那道令人沮丧的鸿沟，落到 async Rust 这一侧。” -- 从事汽车数字驾驶舱系统的软件工程师

> “我的整体印象其实相当负面。感觉它还没做熟……要想高效使用它，你需要很多很玄的知识，比如 Pin---比如说实话，我根本说不清 Pin 是怎么工作的，对吧？” -- 熟悉 Rust 的研究软件工程师

要让 Rust 提供那种“值得信赖、能帮你攻克新领域的工具”的体验，人们就需要能够把自己对 Rust 的既有预期和知识迁移到那个新领域里去。对于 async 来说，不仅语言特性还不完整（比如 trait 里的 `async fn` 直到去年才可用，而且现在仍有缺口），而且用户原本依赖来“弥合鸿沟”的支持性工具链和生态，在这里也没那么好用了：

> “我当时支持不要用 async，因为那些错误信息实在太难处理了。” -- 桌面应用开发者

> “直到现在，仍然有很多情况是你会想 *那个库看起来很有用，我想用它*，然后你马上就被锁进 tokio-rs 或其他某个运行时里，于是你会想 *这就有点让人失望了，因为我自己也想写一个库，现在却被绑定到某个运行时上了*。” -- 从事 Linux 功能安全的安全系统工程师

> “我们通常用 Rust 写服务，而且我们大量使用 async，因为很多跟数据库以及其他东西交互的库都是 async 的。我们遇到问题的时候，往往是像，嗯，莫名其妙的高 CPU 占用之类的。真正比较直接的排查或诊断方式，只有像：*好吧，我去挂上 GDB，看看所有线程都在干什么*。GDB--当然，这显然不是 Rust 的错--但 GDB 不是一个特别容易用的工具，尤其是在大型应用里。[..] 而到了 async，这会更难，因为你看不到你的代码在运行，它其实这会儿只是躺在堆上。早期的时候，我其实根本没意识到事情是这样的。” -- 某家同时使用 Rust 和 Python 的公司里的资深 Rust 开发者

Async 重要到值得单独深入研究。我们的研究揭示了大量挫败感，但我们挖得还不够深，无法给出更具体的洞见。这会是未来用户研究团队很适合承担的一项任务（正如[我们第一篇文章](https://blog.rust-lang.org/2025/12/03/lessons-learned-from-the-rust-vision-doc-process/)中提议的那样）。

### 例子：crates.io 上丰富的 crate 是关键助力，但也可能成为障碍

前面提到过，Rust 的可扩展性是它实现通用性的原因之一。像可重载运算符、trait 和宏这样的机制，让库能够为开发者创造丰富的使用体验；精简的标准库加上易用的包管理，又鼓励形成一个覆盖常见与小众需求的丰富 crate 生态。不过，尤其是在人们刚开始接触 Rust 的时候，这种 *可扩展性* 也可能以牺牲 *支持性* 为代价，当“选择的暴政”开始压得人喘不过气时就是如此：

> “要用哪些 crate，其实很难发现。关于某些具体事情该用什么 crate，存在一层隐性的知识，你只能通过经验和踩坑慢慢积累。每个人都得自己做全部研究。” -- 从事开发者框架工作的 Web 开发者兼大会演讲者

> “Crates.io 确实提供了一些做这些决策所需的元数据，但它并不是一个一站式答案，对吧？不是说你去 crates.io 问一句‘我想实现 X，我该用哪个库’---它就会直接告诉你答案。” -- 研究软件工程师

Rust 组织在历史上一直不太愿意在生态里“钦点”某些 crate。但现实是，有些 crate 的存在感无处不在。这对新用户来说尤其难以应对：

> “教程里用的是 `Result<Box<dyn Error>>` --但别人都不用。所有人都在用 anyhow-result……我一开始也是跟着用 result 那套，但我找到的资料示例代码全都在用 anyhow。这就有点不匹配，我不知道自己该怎么做。” -- 从事数据科学平台工作的软件工程师

> “关于该用哪些 3P crate，并没有清晰、成文的共识。[..] 有时候真的很不清楚---比如 CBOR 该用哪个 crate？[..] 也不容易看出来哪些 crate 还在积极维护。[..] crates.io 上有这么多 crate，这本身就带来了一点风险。” -- 某大型科技公司的 Rust 团队

## 建议

### 梳理 Rust 的设计目标，并将其纳入我们的流程

我们建议创建一份 RFC，明确我们在 Rust 上工作时所追求的目标。这份 RFC 应该覆盖使用 Rust 的整体体验（语言、工具和库）。这份 RFC 可以由拟议中的用户研究团队撰写，但应该由谁来接受它还不清楚--也许是用户研究团队自己，也许是领导委员会。

这篇文章指出，Rust 真正“赋能的魔力”来自同时实现多种不同属性--可靠性、效率、底层控制力、支持性，等等。若能有一份这些价值的权威清单，让社区在讨论时可以共同引用，并在评估 RFC 或其他设计提案时加以使用，那会很有价值。

在这项工作上，过去已经有不少可供借鉴的方法（比如 [Tyler Mandry 的这篇文章](https://tmandry.gitlab.io/blog/posts/the-main-thing/)、[Rustacean Principles](https://smallcultfollowing.com/babysteps/blog/2021/09/08/rustacean-principles/) 或 [Rust Design Axioms](https://smallcultfollowing.com/babysteps/blog/2023/12/07/rust-design-axioms/)）。我们的研究带来的一个洞见是：我们不需要界定哪些价值“最重要”。我们已经看到，要让 Rust 真正成立，它必须同时实现**所有**这些因素。与其给它们排名，不如描述在以下情况下的感受：

- **没做到**（太少）
- **做到刚刚好**（最佳状态）
- **做过头了**（太多）

这种“刚刚好”的框架有助于人们识别自己所处的位置并及时修正，而不会制造出虚假的等级秩序。

### 加倍投入可扩展性

我们建议把 **加倍投入可扩展性** 作为核心策略。Rust 的可扩展性--trait、宏、运算符重载--一直是其通用性的关键。但这种可扩展性目前主要集中在某些领域：类型系统和早期阶段的过程宏。我们应该把它扩展到 **支持性接口**（让 crate 提供更好的诊断和引导）以及 **编译工作流**（让 crate 能在构建流程的更多阶段参与进来）。

Rust 的可扩展性，是 Rust 得以实现通用性的重要原因，而这种通用性又是人们热爱 Rust 的重要原因。借助过程宏、trait 系统和借用检查器这样的机制，Rust crate 能够暴露出高层次、优雅的接口，并在编译后落到高效的机器码上。做到最好时，这种体验甚至会带一点魔法感。

但遗憾的是，Rust 虽然给 crate 提供了构建安全、高效抽象的好工具，却没有提供足够的工具来构建 **支持性** 抽象。在 Rust 内建语言概念之中，我们投入了很多精力去打造有效的错误信息，帮助用户走向成功；我们随编译器发布 lint，用于捕捉常见错误或强制执行重要约定。但 crate 并不能享受到这些。像 [RFC #3368](https://rust-lang.github.io/rfcs/3368-diagnostic-attribute-namespace.html) 这样引入 diagnostic 命名空间和 `#[diagnostic::on_unimplemented]` 的 RFC，表明 Rust 已经开始朝这个方向推进。我们应该继续下去，并寻找进一步拓展的机会，尤其是针对那些经常构造出类 DSL 接口的 proc-macro。

可扩展性的另一个主要挑战与构建系统和后端有关。Rust 当前的可扩展机制（如 build.rs、proc-macros）主要聚焦在编译流程的 *早期阶段*。但 Rust 的许多扩展方向--从互操作到定理证明，再到 GPU 编程和分布式系统--都会受益于能够接入编译流程的其他阶段。[Stable MIR](https://github.com/rust-lang/project-stable-mir) 项目以及 [build-std project goal](https://rust-lang.github.io/rust-project-goals/2025h2/build-std.html?highlight=std#build-std) 都是这类工作的例子。

加倍投入可扩展性，不仅会让今天的 Rust 更易于使用，也会使 Rust 能够进入并支持新的领域。尤其是安全关键应用，需要大量定制 lint 和工具链来满足相关标准。编译器可扩展性能够让 Rust 以更一般化的方式支持这些小众需求。

### 帮助用户在 Rust 生态中建立方向感

我们建议寻找方法，帮助用户浏览 crates.io 生态。今天的惯用 Rust，在从错误处理到 async 运行时的方方面面都依赖自定义 crate。倚重生态有助于 Rust 扩展到更多领域，也让创新的新做法有机会被发现。但对刚起步的人来说，到底该用哪些 crate 是一个非常现实的障碍。Rust 组织一直维持着审慎中立的立场，这很好，但也意味着人们没有地方可以得到关于一套好用“起步 crate”的建议。

这里的正确解法并不明显。扩充标准库可能会切断进一步实验的空间；“钦点”crate 又会带来政治风险。但正确解法难找，并不意味着我们应该忽视这个问题。Rust 一直有用创造性办法重新处理旧有权衡的历史，我们也应该把这种能量投入到这个问题上。

解决方案的一部分，是让库与库之间更容易互操作。这可以表现为增加关键的互操作 trait（尤其是 async 方面），也可以是认可一些标准构件（例如 `http` crate，它为 HTTP 库提供类型定义）。对一致性规则（coherence rules）的修改也会有所帮助，因为当前规则不允许生态里引入一个新的互操作 trait，再逐步被采用。

## 结语

总结一下，这篇文章的要点如下：

* 人们热爱 Rust 的，是它让他们有能力去解决棘手的问题、进入新的领域。这并不是任何单一特性的结果，而是多种因素之间精心平衡的产物；只要其中任何一点受损，这门语言都会明显受影响。
* 为了帮助 Rust 继续在不同领域和不同使用层级之间扩展，我们提出三项建议
  * 梳理并描述 Rust 的设计目标，并将其纳入我们的流程，帮助确保未来的语言设计者和更广泛的生态都能遵守它们。
  * 加倍投入可扩展性，让 crate 能够影响开发体验和编译流水线。
  * 帮助用户在 crates.io 生态中找到方向，并实现更顺畅的互操作
