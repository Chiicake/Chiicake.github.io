# Go 服务限流的几种落地模式

限流不是为了“让接口慢一点”，而是为了在容量有限时有选择地拒绝请求，从而保护更重要的流量和更昂贵的依赖。

在 Go 服务里，限流通常不会只做一层。真正稳定的方案，往往是边缘、服务节点和共享状态三层协作。

## 1. 先回答三个问题

在写任何代码之前，先把这三个问题说清楚：

1. 你要保护的是 CPU、数据库连接池，还是下游第三方配额？
2. 你要按什么维度限流，是用户、IP、租户还是 API 路径？
3. 超限后你要返回什么，是立即拒绝、排队等待还是降级到缓存结果？

这三个答案会直接决定你用哪种算法，以及应该把限流放在哪一层。

## 2. 常见算法如何取舍

### 令牌桶

令牌桶适合允许短时突发的流量模型。只要桶里还有令牌，请求就可以通过；桶空了就拒绝或等待。

它的优势是实现简单、吞吐高，特别适合接口级别的基础保护。

### 滑动窗口

滑动窗口更适合“统计意义上必须严格”的限制，比如每分钟只能调用多少次。

它比固定窗口更平滑，不容易在窗口边界出现突刺，但实现复杂度和存储开销更高。

### 漏桶或队列

如果你的目标不是直接拒绝，而是把流量整形成稳定速率输出，那么漏桶和任务队列会更合适。典型场景是发短信、调用第三方支付或批量写库。

## 3. 三层限流的职责划分

### 边缘层

在 API Gateway 或 Ingress 层做粗粒度限流，优先拦住明显异常流量，比如：

- 单 IP 突发洪峰；
- 明显超频的匿名访问；
- 低价值路径的大量扫描请求。

边缘层的目标是低成本挡住噪音。

### 服务节点层

在 Go 应用本地做轻量令牌桶，保护当前进程的 goroutine、CPU 和连接池。这一层不依赖网络往返，响应最快。

它很适合：

- 每个实例的并发保护；
- 访问某个昂贵下游前的本地闸门；
- 热点接口的快速熔断。

### 共享状态层

当限流维度需要跨实例一致时，就必须引入 Redis 或其它共享状态存储，例如“同一租户每分钟只能写入 600 次”。

这一层适合做：

- 多实例全局配额；
- 多租户公平性；
- 需要审计和计费的请求计数。

## 4. Redis 方案里的两个细节

很多团队在 Redis 限流里只关注 Lua 脚本是否原子，却忽略了两个更常见的问题。

### 热 Key

如果所有请求都打到同一个租户或同一条热门 API，单个计数 Key 很容易成为瓶颈。需要考虑分片、局部缓存或者把超热点租户单独隔离。

### 失败退化

当 Redis 自己抖动时，服务是选择全部放行、全部拒绝，还是只保护关键路径？这个决策不能留到事故发生时再临场判断。

## 5. 可观测性比算法名字更重要

真正难做的从来不是“写一个令牌桶”，而是知道限流是否打在了正确的位置。

至少要监控：

- 每条规则的命中次数与拒绝次数；
- 被限流请求的用户、租户和路径分布；
- 触发限流前后的下游延迟变化；
- Redis 或网关限流组件本身的错误率。

## 6. 一条实用建议

如果你的服务还没有任何限流，不要一上来就做“全局最优方案”。更稳妥的路径是：

1. 先在边缘层做粗限流；
2. 再在最贵的几个接口上加本地令牌桶；
3. 只有当业务明确需要跨实例公平性时，再引入 Redis 规则。

限流的本质不是算法炫技，而是容量治理。谁最贵、谁最脆弱、谁最值得保护，应该先被回答，然后才轮到代码。
