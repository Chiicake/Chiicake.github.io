# JWT 签名校验中的常见陷阱

JWT 很容易给人一种“只是把几段 Base64 字符串拼起来”的错觉，但真正困难的部分从来不是编码，而是你如何校验签名、绑定上下文，并在密钥轮换时保持兼容。

在生产系统里，JWT 出问题通常不是因为库不会验签，而是因为边界条件被默默忽略了。下面这几类问题最常见。

## 1. 不要把 `alg` 当作客户端建议

JWT Header 里的 `alg` 字段必须被服务端当作“待验证输入”，而不是“可信配置”。

如果后端直接根据 Token Header 决定验签方式，就可能出现两类严重问题：

1. 攻击者把算法从 `RS256` 改成 `HS256`，诱导服务端错误地把公钥当成 HMAC 密钥。
2. 服务端默认接受 `none` 或其它本不该启用的算法，导致签名形同虚设。

### 正确做法

- 在服务端配置白名单，只允许明确启用的算法。
- 按签发方、租户或应用维度绑定算法，不允许同一验证入口“自动猜测”。
- 不要混用对称密钥和非对称公钥的验证路径。

## 2. 验签成功不等于 Token 可用

很多系统在签名校验通过后，就直接把 JWT 当作可信身份。这是不完整的。

签名只说明 Token 的内容来自某个私钥对应的签发方，不代表它适用于当前请求。至少还需要同时校验：

- `iss`：是不是你信任的发行者。
- `aud`：是不是发给当前 API 或前端的令牌。
- `exp` / `nbf`：当前时间是否在有效期内。
- `sub`：主体是否符合你的业务类型。
- `jti`：是否需要做一次性令牌或撤销控制。

### 关于时钟漂移

分布式环境里常见的一个现象是：签发服务和消费服务相差几十秒，导致刚签出来的 Token 被判定为“尚未生效”或“已经过期”。

实务里通常会给 `exp` 和 `nbf` 留一个很小的宽限窗口，但这个窗口应该明确、可观测，并且统一实现，不能由每个服务自己猜。

## 3. `kid` 轮换需要兜底策略

只要你开始做密钥轮换，`kid` 就会变成关键字段。它的作用是告诉验证方“应该从哪把钥匙里取对应密钥”。

但是很多实现把 `kid` 当作任意文件路径、数据库主键甚至 URL 参数使用，这会引入新的攻击面。

### 轮换时要注意三件事

1. `kid` 只能映射到受控密钥集合，不能直接拼接文件路径。
2. 新旧密钥需要有重叠窗口，否则老 Token 会在轮换瞬间全部失效。
3. 如果你使用远程 JWK Set，缓存策略和失败兜底要设计清楚，否则流量高峰时会把认证依赖打穿。

一个稳妥的方案通常是：

- 进程内缓存最近一批公钥；
- 后台异步刷新 JWK；
- 当 `kid` 未命中时只做一次受控回源，不要所有请求同时击穿。

## 4. 多租户场景下要隔离验证上下文

如果系统服务多个租户、多个应用或多个环境，最危险的设计是“只要签名合法就放行”。

同一套验证逻辑如果同时接受测试环境、生产环境、管理后台和用户前台的 Token，很容易把本来属于 A 场景的令牌错误地用到 B 场景。

### 建议的隔离方式

- 按租户或应用绑定独立的 `issuer` 和 `audience`。
- 为不同用途的 Token 使用不同密钥集合。
- 在验证入口就区分管理端、用户端、服务间调用，不要等到业务层再做模糊判断。

## 5. 生产环境检查清单

如果你要审查一套 JWT 实现，我通常会先看下面这份清单：

1. 算法是否服务端固定，而不是从 Header 直接接受。
2. 是否同时校验 `iss`、`aud`、`exp`、`nbf`。
3. `kid` 是否只映射到受控密钥集合。
4. 密钥轮换时是否有缓存、重叠窗口和回源限流。
5. 是否区分不同租户、环境和调用场景。

JWT 不是“有签名就安全”的格式，而是一套需要上下文约束的令牌协议。真正可靠的实现，重点永远在签名之外的这些边界。
